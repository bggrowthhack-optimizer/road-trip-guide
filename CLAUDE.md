# CLAUDE.md — правила для Claude Code

## Архитектура

- **Источник истины по архитектуре**: [`ARCHITECTURE.md`](ARCHITECTURE.md) —
  продуктовая идея, порядок сценариев, принцип Wikipedia-first, стратегия
  кеширования, стек. При любом архитектурном решении (смена источника
  данных, смена стратегии кеша, новый сценарий) — сверяйся с этим файлом
  и обновляй его, если решение меняет описанный там подход.

## Промпты

- Промпты для LLM хранятся отдельными `.md`-файлами рядом с сервисом,
  который их использует (например `backend/app/point_summary_prompt.md`),
  не строками внутри Python/TypeScript кода. Так их проще версионировать
  и редактировать без пересборки.

## Кеш контента

- Любой сгенерированный контент (карточка точки, текст для аудио, сам
  аудиофайл) должен сохраняться в кеш с привязкой к устойчивому
  географическому идентификатору (координаты или ID статьи Википедии),
  а не привязываться к конкретному маршруту или пользователю. Это
  единственный способ, которым система становится дешевле и быстрее со
  временем — см. `ARCHITECTURE.md`, раздел "Экономика".
- Перед тем как писать код генерации новой точки — сначала проверка кеша.
  Не наоборот.

## Источники данных

- По умолчанию — Wikipedia (geosearch + содержимое статьи) как основной
  источник кандидатов и первого черновика факта. Тяжёлый веб-поиск по
  многим источникам на каждую точку (как в архивной версии) не делаем —
  это то, от чего мы сознательно ушли после ручного прогона первой
  версии (см. `archive/road-trip-guide-v1/PROCESS.md`).
- **Уточнение по итогам второй проверки**: веб-поиск (минимум один
  целевой запрос) — **обязателен для каждой точки**, а не только для
  тех, что выглядят "интересными" по косвенным признакам. Population,
  длина статьи в Wikipedia и подобные метрики **не предсказывают**,
  найдётся ли легенда/личность/драма — на практике самые "пустые" стабы
  (Мирный, Малаевка, Озёрки) после одного поиска дали одни из лучших
  историй маршрута Ульяновск–Самара. Не решать заранее "это маленькое
  село, там скорее всего нечего искать" — сначала искать, потом решать,
  насколько длинным получится итоговый текст.
- Хорошие паттерны запроса: `[название] [район] история легенды
  название` и `[название] [район] история известные люди`. У многих
  районов есть архивные краеведческие сайты администрации с
  историческими справками по каждому селу (пример: `cherdakli.com`) —
  проверять их, не только Wikipedia и её зеркала.
- Если после честной попытки поиска ничего не нашлось — короткая
  честная карточка (только то, что даёт Wikipedia), без выдумывания.
  Это ожидаемый, а не провальный результат для части маленьких сёл.

## Формат резюме

- Целевая длина — не короткая карточка (100-150 слов), а полноценный
  рассказ на 350-450 слов (~2-3 минуты аудио) — по прямой обратной связи
  пользователя ("возникает желание побольше слушать"). Длина
  ограничивается фактическим объёмом найденного материала, не наоборот:
  если после честного поиска нашлось мало — короткая карточка это ок
  (см. "Источники данных" выше), просто не должна быть дефолтным
  результатом без попытки найти больше.
- Пять слоёв повествования, не все обязательны, но в этом порядке
  дают драматургию (эталонный пример — карточка "Чердаклы"):
  1. **Хук** — первая фраза цепляет (легенда/странность/неожиданный
     факт), не сухая дата основания.
  2. **Происхождение + легенда одной историей** — не как два отдельных
     факта, а сплетены: кто/когда/почему назвали именно так.
  3. **Личность с конкретикой** — не просто имя, а что человек делал,
     почему оказался здесь, в чём его роль.
  4. **Поворотный момент** — историческая драма, которая придаёт вес
     (война, затопление, переименование, восстание и т.п.).
  5. **Настоящее время** — что реально можно увидеть/заметить сегодня,
     проезжая мимо — даёт слушателю физическую зацепку, а не только факт.
- Экономика этого решения — см. `ARCHITECTURE.md`, раздел "Экономика":
  более длинный текст и обязательный поиск дороже сами по себе, но это
  разовая стоимость на конкретную точку на карте, а не на каждый
  запрос/пользователя — не бояться увеличивать глубину контента на
  этом основании.

## Архив

- `archive/road-trip-guide-v1/` — первая, полностью ручная версия проекта
  (статический HTML-гид по конкретному маршруту Кисловодск–Самара).
  Не рабочий код для переиспользования как есть, но ценный источник
  проверенных решений: формат аудио-рассказа (не диалог), механика
  подкаст-плеера (перемотка назад, переход между главами), структура
  вложенности точек интереса. Смотреть туда при проектировании
  соответствующих кусков нового сервиса, не удалять.
- **Демо для обратной связи**: та же статика опубликована через GitHub
  Pages из отдельной ветки `gh-pages` (не `main`) —
  https://bggrowthhack-optimizer.github.io/roadcast/. Если контент в
  `archive/road-trip-guide-v1/` меняется и демо нужно обновить — скопировать
  актуальные `index.html`/`audio/` в отдельный воркtree на ветке `gh-pages`
  и запушить именно туда; `main` это не затрагивает.

## Git

- Коммитить значимые файлы (код, миграции, конфиги, документацию).
  Временные/диагностические скрипты и артефакты сборки — не коммитить.
- Новые коммиты, не amend, если явно не попросили иначе.
- **Ветки**: пока нет стабильной версии, важной для кого-то кроме автора —
  работать напрямую в `main`. Как только появляется рабочая версия,
  которую не хочется сломать (кто-то тестирует MVP, есть демо, на которое
  можно ссылаться) — переходить на одну ветку на одну задачу, слияние в
  `main` через Pull Request (даже если ревьюер — сам пользователь).
  Проверка себя перед началом работы: «если это сломается на пару часов,
  кому-то будет больно?» — если да, заводить ветку, не пушить прямо в
  `main`.
- **Перед коммитом — показать пользователю, что именно меняется** (диф
  или чёткое краткое summary), а не коммитить молча "на веру". Это не
  замена ревью пользователем, а обязательный шаг перед любым коммитом.

## Ведение задач (TaskCreate)

- Для любой работы из 3+ шагов, или растянутой во времени (несколько
  фоновых агентов, несколько этапов пайплайна) — вести список задач через
  `TaskCreate`/`TaskUpdate`, отмечать прогресс по ходу выполнения, а не
  держать план в голове. Пользователь — продакт с перерывом в разработке
  на 15 лет, ему важно видеть прогресс явно, а не догадываться по
  переписке, что уже сделано.

## Общий стиль работы с пользователем

- Пользователь предпочитает, чтобы после согласования архитектуры/решения
  агент действовал автономно, без лишних промежуточных вопросов, и
  показывал результат целиком, а не по шагам с уточнениями на каждом.
- Если пользователь сомневается в масштабе задачи или сроках — честно
  обозначать реальный объём работы и уместные компромиссы (пример:
  переход с ручного многочасового веб-исследования на Wikipedia-first
  был принят именно потому, что агент предложил его как более быстрый и
  дешёвый путь, а не потому что попросили).
