# CLAUDE.md — правила для Claude Code

## Архитектура

- **Источник истины по архитектуре**: [`ARCHITECTURE.md`](ARCHITECTURE.md) —
  продуктовая идея, порядок сценариев, принцип Wikipedia-first, стратегия
  кеширования, стек. При любом архитектурном решении (смена источника
  данных, смена стратегии кеша, новый сценарий) — сверяйся с этим файлом
  и обновляй его, если решение меняет описанный там подход.

## Промпты

- Промпты для LLM хранятся отдельными `.md`-файлами рядом с сервисом,
  который их использует (например `backend/app/point_summary_prompt.md`),
  не строками внутри Python/TypeScript кода. Так их проще версионировать
  и редактировать без пересборки.

## Кеш контента

- Любой сгенерированный контент (карточка точки, текст для аудио, сам
  аудиофайл) должен сохраняться в кеш с привязкой к устойчивому
  географическому идентификатору (координаты или ID статьи Википедии),
  а не привязываться к конкретному маршруту или пользователю. Это
  единственный способ, которым система становится дешевле и быстрее со
  временем — см. `ARCHITECTURE.md`, раздел "Экономика".
- Перед тем как писать код генерации новой точки — сначала проверка кеша.
  Не наоборот.

## Источники данных

- По умолчанию — Wikipedia (geosearch + содержимое статьи). Веб-поиск по
  произвольным источникам — только явный fallback для точек без статьи
  или с очень коротким стабом. Не делать наоборот (веб-поиск по
  умолчанию) — это то, от чего мы сознательно ушли после ручного прогона
  первой версии (см. `archive/road-trip-guide-v1/PROCESS.md`).

## Архив

- `archive/road-trip-guide-v1/` — первая, полностью ручная версия проекта
  (статический HTML-гид по конкретному маршруту Кисловодск–Самара).
  Не рабочий код для переиспользования как есть, но ценный источник
  проверенных решений: формат аудио-рассказа (не диалог), механика
  подкаст-плеера (перемотка назад, переход между главами), структура
  вложенности точек интереса. Смотреть туда при проектировании
  соответствующих кусков нового сервиса, не удалять.
- **Демо для обратной связи**: та же статика опубликована через GitHub
  Pages из отдельной ветки `gh-pages` (не `main`) —
  https://bggrowthhack-optimizer.github.io/roadcast/. Если контент в
  `archive/road-trip-guide-v1/` меняется и демо нужно обновить — скопировать
  актуальные `index.html`/`audio/` в отдельный воркtree на ветке `gh-pages`
  и запушить именно туда; `main` это не затрагивает.

## Git

- Коммитить значимые файлы (код, миграции, конфиги, документацию).
  Временные/диагностические скрипты и артефакты сборки — не коммитить.
- Новые коммиты, не amend, если явно не попросили иначе.

## Общий стиль работы с пользователем

- Пользователь предпочитает, чтобы после согласования архитектуры/решения
  агент действовал автономно, без лишних промежуточных вопросов, и
  показывал результат целиком, а не по шагам с уточнениями на каждом.
- Если пользователь сомневается в масштабе задачи или сроках — честно
  обозначать реальный объём работы и уместные компромиссы (пример:
  переход с ручного многочасового веб-исследования на Wikipedia-first
  был принят именно потому, что агент предложил его как более быстрый и
  дешёвый путь, а не потому что попросили).
