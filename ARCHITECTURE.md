# Roadcast — архитектура

## Продукт одним предложением

Пользователь называет маршрут (старт, финиш, опционально — промежуточные
точки) и тип перемещения (машина/поезд/прогулка/самолёт) — сервис собирает
«книгу путешествия»: связную историю мест по пути, которую можно читать или
слушать.

Проверено вручную на реальной поездке (Кисловодск → Элиста → Волгоград →
Самара, см. `archive/road-trip-guide-v1/`) — общая механика работает и
нравится пользователю. Задача теперь — превратить ручной процесс в сервис.

---

## Порядок сценариев

1. **Машина** — первый MVP. Линейный маршрут, чёткая цепочка населённых
   пунктов и объектов вдоль дороги. Уже проверено вручную.
2. **Поезд** — та же механика (линейный маршрут + промежуточные пункты),
   почти бесплатное расширение после (1) — просто другая трасса/траектория.
3. **Прогулка по городу** — та же идея «точки вдоль линии», но другой
   масштаб: улицы и здания вместо городов, нужна более плотная база POI.
4. **Самолёт** — принципиально другой контент: нет промежуточных остановок
   на земле. Нужен отдельный сценарий повествования (города вылета/прилёта,
   что видно с высоты, история маршрута) поверх той же инфраструктуры
   хранения и TTS, а не расширение линейного движка. Делать последним.

---

## Ключевой архитектурный принцип: Wikipedia-first

**Наблюдение с реальной поездки**: пользователю в дороге хватало информации
из Википедии почти всегда; выход за её пределы — редкое исключение, а не
правило (это отличается от того, как собирался `road-trip-guide-v1` — там
на каждую точку гонялся тяжёлый агент с десятками веб-поисков по разным
источникам, что заняло часы на один маршрут).

Из этого — главный принцип экономики и скорости системы:

1. **Поиск точек и контента — через Wikipedia API, а не общий веб-поиск.**
   - `list=geosearch` (MediaWiki API) — статьи в радиусе N км от точки на
     маршруте → это одновременно и источник кандидатов в населённые пункты
     /объекты, и источник контента. Один API закрывает обе задачи, которые
     раньше решались раздельно (маршрутизация населённых пунктов + отдельно
     веб-поиск фактов).
   - Контент статьи → LLM нужен не для *поиска* фактов, а только для (а)
     сжатия статьи в карточку, (б) переписывания в связный рассказ для
     аудио, (в) сшивки нескольких точек в единое повествование маршрута.
2. **Веб-поиск — явный fallback, а не путь по умолчанию.** Включается,
   когда: точки нет в Википедии, статья — стаб на 1-2 предложения, либо
   пользователь прямо запросил более глубокое погружение. Ровно то, что мы
   видели вручную с мелкими сёлами (Подкумок, Оленье и т.п.) — честная
   короткая справка вместо выдумывания.
3. Это не отменяет правило перекрёстной проверки для спорных случаев
   (тёзки населённых пунктов, административные привязки) — но делает его
   исключением при генерации, а не стандартной процедурой на каждую точку.

---

## Маршрут и точки: реальные геоданные, не веб-поиск

Определение маршрута вручную (веб-поиск + сверка через грузоперевозочные
сервисы) заняло часы на один маршрут — неприемлемо для сервиса с откликом
в единицы минут. Переход на геоданные:

- **Геометрия маршрута** — через routing API (Яндекс.Карты API / OSRM /
  GraphHopper) вместо текстового описания дорог.
- **Точки вдоль маршрута** — Wikipedia geosearch вдоль линии маршрута
  (буфер N км) как основной источник; OpenStreetMap/Overpass — как
  расширение, если нужна более плотная сетка объектов (особенно для
  сценария «прогулка по городу», где плотность POI на километр значительно
  выше, чем для автомобильного маршрута).

---

## Экономика: кеш-база знаний, а не генерация с нуля на каждый запрос

Центральный приём, который делает сервис одновременно быстрым и дешёвым:

- Контент и аудио по каждой точке (например, «Кисловодск», «гора Кольцо»)
  генерируются один раз и сохраняются в базе с геопривязкой (координаты
  или устойчивый ID места/статьи Википедии).
- Следующий пользователь, чей маршрут проходит через ту же точку, получает
  готовый контент мгновенно — без повторного обращения к LLM или TTS.
- Только точки, которых ещё нет в базе, проходят полный цикл генерации.
  Система «богатеет» со временem: чем больше маршрутов обработано, тем
  выше процент карточек, отдаваемых из кеша, и тем быстрее и дешевле новый
  запрос.
- Практическое следствие для UX: генерация — асинхронная задача с
  прогресс-баром («собираем историю…»), а не блокирующий запрос. Для
  маршрута с большой долей закешированных точек ответ может быть почти
  мгновенным.

## Аудио

Продакшен не может полагаться на локальный `say` (macOS) — нужен
облачный TTS с реальным API (кандидаты: OpenAI TTS, ElevenLabs, Google/
Azure). Это статья расходов сервиса, но она возникает только при первой
генерации точки — дальше аудиофайл переиспользуется из кеша/объектного
хранилища так же, как текстовый контент.

Формат — связный рассказ одним голосом (не диалог, см. `archive/
road-trip-guide-v1/PROCESS.md` — диалог двух ведущих пробовали и
отклонили, звучал рвано). Один аудиофайл на точку/объект, плеер собирает
их в порядке маршрута с возможностью перемотки назад и перехода между
главами — эта часть уже отработана в архивной версии, механику плеера
можно переносить как есть.

## Технологический стек

Ориентируемся на стек соседнего проекта команды (`optics_summary`) —
он уже проверен, и на нём есть опыт разработчиков:

- **Backend**: FastAPI (Python, асинхронный), Uvicorn, Pydantic для схем
- **Frontend**: Next.js 15 (App Router), React, TypeScript, Tailwind CSS,
  shadcn/ui
- **LLM-доступ**: через тот же прокси-провайдер, что и в `optics_summary`
  (`polza.ai`, OpenAI-совместимый API к Claude) — переиспользуем готовую
  интеграцию вместо того, чтобы придумывать новую
- **База данных**: PostgreSQL (кеш точек/контента, геопривязка — со
  временем может понадобиться PostGIS для эффективных geo-запросов)
- **Хранилище аудио**: объектное хранилище (S3-совместимое)
- **Промпты** — отдельными `.md`-файлами рядом с сервисами, не строками
  в коде (паттерн `optics_summary`: `doctor_analysis_prompt.md` и т.п.) —
  проще итерировать и версионировать независимо от кода

## MVP: что делаем первым

Сценарий 1 (машина) целиком, минимальный сквозной путь:

1. Форма: старт, финиш, тип транспорта (пока только «машина»)
2. Routing API → геометрия маршрута
3. Wikipedia geosearch вдоль маршрута → список точек-кандидатов
4. Для каждой точки: есть в кеше? → отдаём; нет → LLM собирает карточку
   из статьи Википедии + пишет текст для аудио → TTS → сохраняем в кеш
5. Сборка «книги»: фронтенд показывает карточки в порядке маршрута +
   плеер (логика уже есть в архивной версии, переносим)
6. Один живой прогон целиком, на реальном маршруте, прежде чем
   переходить к сценарию 2

## Открытые вопросы (требуют решения по ходу, не блокируют старт)

- Точный выбор routing API (зависит от бюджета/лимитов — Яндекс.Карты API
  платный для коммерческого использования, нужно уточнить условия;
  сейчас используется бесплатный demo-сервер OSRM, см. "Результаты первой
  проверки" ниже — он не для продакшена, но пока работает для теста)
- Стратегия обновления кеша — что если статья Википедии изменилась
  (TTL на карточку? Ручной триггер обновления?)
- Выбор конкретного облачного TTS-провайдера и голоса

## Результаты первой проверки (Ульяновск → Самара, без кеша, без LLM-ключа)

Первый живой прогон Wikipedia-first пайплайна (`backend/app/route_discovery.py`)
на реальном, заранее не готовившемся маршруте. Контент писал вручную я
(агент) по уже собранным данным — реальный автоматический вызов LLM ещё
не подключён (нет ключа).

**Что подтвердилось:**
- Связка геокодинг → маршрут → Wikipedia geosearch → extracts работает и
  находит содержательные точки на произвольном маршруте, не только на
  заранее подготовленном.
- Формат "связный рассказ на точку" из статьи Википедии получается
  живым даже для многих непримечательных на первый взгляд объектов
  (ТЭЦ, институт, завод) — материала на Wikipedia обычно достаточно.

**Что не подтвердилось сразу и потребовало доработки:**
- **Сырой geosearch рядом с крупными городами возвращает огромный шум.**
  Первый прогон (радиус 7 км, шаг сэмплирования 5 км, без фильтра) дал
  **263 точки** на маршрут в 248 км — в основном отдельные улицы и дома
  Ульяновска и Самары, а не значимые объекты. После эвристического
  фильтра (`backend/app/point_filter.py`: минимальная длина статьи +
  исключение по паттернам в названии — улицы/площади/административные
  единицы) — **26 точек**, разумный список. Фильтр сделан как отдельная
  переключаемая точка (см. комментарий в файле) — при появлении ключа
  Yandex Maps/Cloud туда добавляется классификация через Yandex Geocoder
  (`kind`: locality/street/house) вместо текстовых эвристик.
- **Nominatim (геокодер) недоступен из текущего окружения** — вернул
  `Access denied` при живой проверке. Заменён на Photon (photon.komoot.io,
  тоже бесплатный и без ключа) — отработал надёжно.
- **OSRM demo-сервер требует TLS 1.3**, а системный Python в этом
  окружении линкован со старым LibreSSL, который не может согласовать
  handshake. Рабочий обход — вызывать `curl` подпроцессом именно для
  запроса к OSRM (см. `backend/app/routing_client.py`); при развёртывании
  на другом окружении (например, с современным OpenSSL) это ограничение
  может не понадобиться, а сам `get_route` остаётся с прежней сигнатурой.
- **Wikipedia иногда троттлит анонимные запросы** после всплеска трафика
  (поймали 429 на повторных прогонах во время разработки) — добавлен
  retry с экспоненциальной паузой и уважением `Retry-After`
  (`backend/app/wikipedia_client.py`).
- Несколько точек оказались дублирующими друг друга по сути (три статьи
  об одном и том же университете Самары до и после объединения в 2015
  году) — не критично для одного прогона, но при масштабировании стоит
  подумать про дедуп не только по `pageid`, но и по смысловой близости.

## Результаты второй проверки: населённые пункты vs достопримечательности,
## глубина резюме

После первой проверки пользователь указал на главную проблему: 26 точек
почти целиком описывали объекты в двух крупных городах (Ульяновск,
Самара), а не сёла и посёлки по пути — ровно то, что интереснее всего в
дороге. Причина — `MIN_EXTRACT_LENGTH = 400` в фильтре отбрасывал именно
короткие стабы, которыми почти всегда являются статьи о маленьких сёлах.

**Решение — классификация вместо единого фильтра**
(`backend/app/point_filter.py`, `is_settlement()`): у статей о населённых
пунктах в русской Википедии есть надёжный признак — категория с
паттерном "Населённые пункты ..." или "НП:Статус:...". Проверено вживую:
у "Малаевка (Ульяновская область)" такая категория есть, у "Троицкий
собор (Симбирск)" — нет. Settlement-статьи теперь сохраняются всегда,
без фильтра по длине; landmark-статьи по-прежнему фильтруются
эвристиками. Landmark привязывается к ближайшему settlement по позиции
вдоль маршрута (вложенность, как в архивной версии).

Проверенные и отклонённые альтернативы для определения "что такое
населённый пункт": Overpass API (оба публичных инстанса недоступны из
этого окружения — 406/таймаут, тот же паттерн блокировки, что раньше
был у Nominatim); Photon reverse-геокодинг с фильтром по слою `city`
(рабочий вариант, но требует лишнего внешнего сервиса в критическом
пути) — категории Wikipedia оказались проще и не требуют новых
источников данных.

**Результат**: с классификацией маршрут Ульяновск→Самара даёт **95**
населённых пунктов (вместо 26 landmark-ориентированных точек) — почти
через каждые 2.5 км встречается сёло/посёлок, что и требовалось.

**Формат и глубина резюме — новый принцип, не предсказывать заранее.**
Изначально резюме писались по остаточному принципу: если статья Wikipedia
короткая — короткое и резюме. Пользователь указал, что это неверно:
"пустые" на вид стабы (Мирный, Малаевка, Озёрки) после одного
целевого веб-поиска дали одни из лучших историй маршрута (переименования,
эсеровский штаб в церкви, путаница с одноимёнными деревнями). Значит,
длина статьи Wikipedia и размер села — плохой предиктор "интересности".

Правило теперь: **обязательный минимум один веб-поиск на каждую точку**,
независимо от размера/длины стаба; глубина итогового текста определяется
по факту находок, а не предсказывается заранее (подробный формат — см.
`CLAUDE.md`, раздел "Формат резюме": пять слоёв — хук, происхождение с
легендой, личность с деталями, поворотный момент, что видно сегодня;
целевая длина 350-450 слов вместо изначальных 100-150).

**Экономика более глубокого подхода**: дороже не обслуживание одного
пользовательского запроса, а разовое "открытие" новой точки на карте —
а она попадает в кеш и дальше бесплатно переиспользуется всеми будущими
маршрутами через это же место (см. "Экономика" выше). Порядок величины:
рост объёма текста в ~2.5-3× даёт пропорциональный, но абсолютно
небольшой рост стоимости LLM-генерации и TTS на точку; обязательный
веб-поиск через типичный поисковый API — доли цента на запрос. Ни то,
ни другое не масштабируется с числом пользователей — только с числом
уникальных новых точек, а оно ограничено и быстро "прогревается" для
популярных маршрутов.
