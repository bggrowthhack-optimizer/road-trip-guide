# Шаблон промпта для исследовательского агента (Этап 3 из PROCESS.md)

Используется для каждого плеча маршрута отдельно. Подставить `{{...}}`
и передать агенту с доступом к веб-поиску (WebSearch/WebFetch).

---

```
Собери краткую, но содержательную справочную информацию для туристического
HTML-путеводителя (маршрут {{ПОЛНЫЙ МАРШРУТ ОДНОЙ СТРОКОЙ}}). Это плечо
{{N}} маршрута: {{НАЗВАНИЕ ПЛЕЧА, НАПРИМЕР "Александровское → Благодарный"}}.
Используй веб-поиск (WebSearch/WebFetch), приоритет — авторитетные источники
(Википедия, официальные сайты администраций районов, краеведческие
материалы, реестр объектов культурного наследия).

Для КАЖДОГО из следующих пунктов подготовь структурированную справку
с четырьмя блоками:
1. **История основания/открытия** — когда и почему возникло
2. **Интересные факты** — 2-4 факта с пояснением
3. **Личности** — известные люди, связанные с местом, с объяснением КАК
   именно они связаны (не просто имя, а что делали/строили/писали/освящали
   и когда). Если личностей достоверно не найдено — прямо напиши
   "известных личностей, связанных с этим местом, не зафиксировано" —
   не выдумывай.
4. **Текст для аудио-рассказа** — 100-150 слов связным текстом (не
   списком!), в разговорном стиле рассказчика в дороге, который
   собирает историю/факты/личности этого пункта в единое повествование.
   Не зачитывать карточку дословно — это отдельный, более живой текст
   для озвучки TTS. Без диалогов, один рассказчик.

СПИСОК ПУНКТОВ:

Населённые пункты:
{{нумерованный список: название (район/регион, статус)}}

Природные достопримечательности:
{{нумерованный список: название (координаты если есть, отдалённость
от трассы)}}

Рукотворные достопримечательности:
{{нумерованный список: название (краткая подсказка, что уже известно —
дата, стиль, архитектор — если есть, чтобы агент проверил, а не искал
с нуля)}}

ВАЖНО:
- Для маленьких сёл/хуторов не выдумывай — если информации мало, дай
  честную краткую справку (год основания, происхождение названия,
  административный статус) без домыслов.
- Указывай реальные даты, имена, проверяй факты. Если что-то не удалось
  точно подтвердить — помечай как "предположительно" или "по данным X".
- Если находишь противоречия между источниками — помечай "по разным
  данным", не сглаживай в одну версию.
- Если по ходу поиска заметишь, что данные в задании (район/регион,
  дата) не совпадают с тем, что говорят источники — прямо укажи это
  как отдельное замечание в конце ответа, не втихую исправляй молча.
- Формат ответа: чистый Markdown, заголовок с номером и названием
  пункта, четыре подзаголовка (История, Факты, Личности, Текст для
  аудио) под каждым.
- Пиши по-русски. 80-150 слов на пункт для рядовых объектов, до 200 —
  для значимых (столицы регионов, крупнейшие храмы и т.п.).

Верни финальный markdown-документ со всеми {{ИТОГО}} пунктами.
```

---

## После получения ответа от агента

1. Просканировать на **специфичные, недавние или удивительные факты**
   (даты после 2023 года, разрушения/пожары/повреждения объектов,
   неожиданные административные привязки) — перепроверить отдельным
   `WebSearch` перед тем, как включать в финальный текст. Модель может
   галлюцинировать правдоподобно звучащие, но неверные детали именно
   в таких местах.
2. Если агент сам пометил расхождение с исходными данными задания
   (см. пункт "ВАЖНО" выше) — решить, нужно ли править таблицу маршрута,
   или просто отразить уточнение в тексте карточки (обычно — второе,
   таблицу маршрута не трогаем без явного запроса пользователя).
3. Сгенерировать аудио по каждому пункту из текста блока "Текст для
   аудио-рассказа":
   `say -v Milena -o {{точка}}.aiff "{{текст}}"`
   `afconvert -f m4af -d aac {{точка}}.aiff audio/{{leg}}/{{точка}}.m4a`
   Один файл на точку (без bitrate-флага у afconvert — падает с
   ошибкой на некоторых версиях macOS).
4. Скомпоновать в HTML по структуре карточек из `PROCESS.md` (Этап 5):
   вложенность, бейдж дистанции, кнопка «слушать» на каждой карточке,
   привязанная к общему подкаст-плееру с главами в порядке маршрута.
